# Current Telegram Dual Room Inbox Pattern

## 목적

텔레그램 개선안 알림을 `카드가 계속 쌓이는 구조`에서 `보고서방 + 체크방` 이중 구조로 바꾸기 위한 기준 문서다.

핵심은 아래 4가지다.

1. 개선안이 여러 개 쌓여도 무엇을 먼저 봐야 하는지 바로 보이게 한다.
2. 버튼이 화면을 과도하게 차지하지 않게 한다.
3. 원문 보고서는 한 번만 보내고 계속 재활용한다.
4. 체크방은 작업 큐처럼 누적되게 만든다.

결론은 아래 구조다.

- `보고서방`
  - 원문 보고서 1회 발송
- `체크방`
  - 누적 체크 인박스 1개 또는 소수의 고정 메시지
- `원문 보기`
  - 체크 항목에서 원문 보고서로 이동
- `승인 / 보류 / 거부`
  - 원문 보고서에서만 처리

## 동작 구조

### 1. 보고서방

보고서방은 `원문 보고서 보관소`다.

여기에는 아래 성격의 메시지가 한 번씩 올라간다.

- 진입 개선안 보고서
- 기다림 개선안 보고서
- 청산 개선안 보고서
- PA8 closeout review 보고서
- PA9 handoff review 보고서

원문 보고서에는 아래가 들어간다.

- 왜 이 제안이 나왔는지
- 어떤 표본과 구간을 보고 있는지
- 기존 대비 얼마나 좋아지는지
- 적용 범위가 어디까지인지
- rollback 기준이 무엇인지
- 승인할 가치가 있는지

즉 `설명은 보고서방`, `업무 큐는 체크방`으로 분리한다.

### 2. 체크방

체크방은 `개선안 인박스`다.

여기에는 새 카드가 계속 새로 올라오지 않는다. 대신 하나의 고정 체크 메시지 또는 소수의 고정 섹션 메시지를 계속 수정한다.

예시:

```text
[개선안 체크 인박스]
미검토 4건 / 보류 1건 / 승인대기 2건 / 완료 3건
[ ] BTC 진입 조정 / 리클레임 재진입 / 표본 27 / 기대개선 +18.4 USD / 원문 보기
[ ] XAU 청산 조정 / 손실 보호 청산 / 표본 11 / 기대개선 +9.2 USD / 원문 보기
[보류] NAS closeout / live-window ready / 원문 보기
[승인대기] PA9 handoff / NAS baseline / 원문 보기
```

즉 새 제안이 올 때마다 새 메시지를 계속 늘리는 것이 아니라 기존 인박스 메시지를 갱신한다.

### 3. 원문 링크

체크방 각 항목은 보고서방의 원문 보고서를 가리키는 링크를 가진다.

사용 흐름은 아래와 같다.

1. 체크방에서 항목 확인
2. `원문 보기` 클릭
3. 보고서방 원문으로 이동
4. 원문에서만 `승인 / 보류 / 거부`

즉 체크방은 `무엇을 볼지 정하는 곳`, 보고서방은 `실제로 읽고 결정하는 곳`이다.

## 왜 이 구조가 좋은가

### 1. 카드 폭탄을 막는다

기존 방식은 개선안 하나마다 큰 카드가 쌓이기 쉽다. 개선안이 많아지면 텔레그램 화면이 빠르게 복잡해진다.

인박스 구조는 아래 장점이 있다.

- 새 개선안이 와도 기존 메시지를 edit하면 된다.
- 목록은 계속 누적되어도 한 화면에서 관리할 수 있다.
- 중요한 것만 위로 보이게 만들 수 있다.

### 2. 무엇을 봐야 하는지 바로 보인다

체크방은 `현재 대기 중인 항목`을 먼저 보여준다.

즉 사용자는 스크롤을 많이 하지 않고도:

- 무엇이 지금 쌓였는지
- 무엇을 먼저 열어봐야 하는지
- 무엇은 이미 보류나 완료인지

를 바로 알 수 있다.

### 3. 승인 버튼 크기를 줄인다

버튼은 체크방 모든 항목에 붙지 않는다. 버튼은 원문 보고서에서만 붙는다.

그래서:

- 체크방은 `작업 큐`
- 보고서방은 `결정 페이지`

가 된다.

## 권장 운영 모델

### 모델 A. 방 2개

- `보고서방`
- `체크방`

가장 단순하다.

### 모델 B. 방 1개 + topic 2개

텔레그램 forum topic 구조를 선호한다면 아래도 가능하다.

- topic 1: `보고서`
- topic 2: `체크`

### 현재 권장

초기 구현은 아래를 권장한다.

- `보고서방` 1개
- `체크방` 1개

이유:

- 구현이 가장 단순하다.
- 링크/메시지 관리가 쉽다.
- 이후 forum topic 구조로도 옮기기 쉽다.

현재 프로젝트처럼 이미 `CFD 체크방`을 forum으로 쓰고 있다면, 이 문서의 `보고서방 / 체크방`은 실제론 아래처럼 읽으면 된다.

- `TELEGRAM_CHECK_CHAT_ID`와 `TELEGRAM_REPORT_CHAT_ID`는 같은 forum chat id 공유
- `TELEGRAM_CHECK_TOPIC_ID`와 `TELEGRAM_REPORT_TOPIC_ID`만 다르게 사용

## 체크방 메시지 구조

### 상단 요약

체크방 고정 메시지 상단에는 아래를 둔다.

- `미검토 n건`
- `보류 n건`
- `승인대기 n건`
- `완료 n건`
- `최종 갱신 시각`

### 항목 한 줄 구조

각 항목은 아래 정도로 요약한다.

- 상태
- 심볼
- 개선 대상
- 주된 이유
- 표본 수
- 기대 개선치
- 원문 링크

예시:

```text
[ ] BTC / 진입 조정 / 리클레임 재진입 / 표본 27 / 기대개선 +18.4 USD / 원문 보기
[보류] NAS / closeout / live-window ready / 표본 1 / 원문 보기
```

### 정렬 규칙

인박스는 아래 순서로 정렬하는 것을 권장한다.

1. `긴급`
2. `미검토`
3. `보류`
4. `승인대기`
5. `완료`

## 보고서방 원문 구조

원문 보고서에는 아래가 들어간다.

- 제목
- 대상 심볼
- 개선 종류
- 관찰 구간
- 비교 결과
- 기대 개선 수치
- 적용 범위
- rollback 기준
- 결정 기한
- 버튼

예시:

```text
[개선안 보고서 | BTC 진입 조정]
대상: BTCUSD
관찰 구간: 최근 4시간

관찰 결과
- 플랫 상태에서 리클레임 재진입 준비 구간에서
- 기존 WAIT보다 ENTER가 더 유리했습니다
- 표본: 27건
- 비교 결과: 승률 +12.5%p, 순손익 +18.4 USD

권장 변경
- 기존: WAIT
- 제안: ENTER
- 범위: BTCUSD / reclaim family / bounded canary
- rollback: worsened rows > 0 또는 runtime match 하락

결정 기한
- 2026-04-11 23:30 KST

[승인] [보류] [거부]
```

## 같은 개선안이 다시 들어왔을 때 규칙

같은 개선안이 다시 계산된다고 해서 매번 원문 보고서를 계속 만들면 안 된다.

### scope_key 기준

같은 `scope_key`면 아래처럼 처리한다.

- 기존 원문이 아직 유효하면 `원문 재사용`
- 체크방 항목만 갱신
- 표본 수, 기대 개선치, 최신 시각 업데이트

### 새 원문이 필요한 경우

아래 같은 경우에만 새 원문을 발행한다.

- 제안 내용 자체가 바뀜
- 기존 WAIT 제안이 ENTER 제안으로 바뀜
- 적용 범위가 바뀜
- rollback 조건이 바뀜
- closeout 단계로 승격됨

## 상태 정의

체크방 항목은 아래 상태를 권장한다.

- `미검토`
- `관찰중`
- `보류`
- `승인대기`
- `적용완료`
- `기각`
- `대체됨`
- `만료`

## env와의 연결

이 구조를 실제로 쓰려면 아래 키가 중요하다.

```env
TELEGRAM_CHECK_CHAT_ID=
TELEGRAM_CHECK_TOPIC_ID=
TELEGRAM_REPORT_CHAT_ID=
TELEGRAM_REPORT_TOPIC_ID=
TELEGRAM_ALLOWED_USER_IDS=
```

규칙:

- 체크방이 일반 방이면 `TELEGRAM_CHECK_TOPIC_ID`는 비운다.
- 보고서방이 일반 방이면 `TELEGRAM_REPORT_TOPIC_ID`는 비운다.
- 승인 버튼을 실제로 쓸 사람은 `TELEGRAM_ALLOWED_USER_IDS`에 넣는다.
